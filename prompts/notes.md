## Project scaffolding expectations

- React app created with a standard tool (CRA, Vite, or similar) and configured for GitHub Pages routing (hash router or `basename`), including a `homepage` setting in `package.json` if needed.[1][2]
- Tailwind fully wired (Tailwind config, PostCSS, `@tailwind base/components/utilities` imported in the main CSS file).[3][4]

## Acceptance criteria for Copilot

Add a short section like this to the spec so Copilot has concrete targets:[5]

- Home page renders a grid of cards from a mocked `apps.json` (hard‑coded initial file) with sort by Newest/Oldest/Highest rated.  
- App detail page reads a single entry from `apps.json` and displays the “Generated by” panel (LLM, agent, timestamps, tokens if present).  
- Suggestion form validates input on the client and logs payload to console (or posts to a stub endpoint) as a first step.  
- Dark/light mode toggle works and persists preference across reloads.  
- Basic registry generator script exists (even if stubbed) that scans `/apps/**/meta.json` and writes `src/data/apps.json`.  
- GitHub Actions workflow file exists that builds the app and (for now) just runs `npm run build` without deploying, so you can wire in the actual deploy action later.[6][1]

## Copilot prompt hints

When you paste this spec into Copilot Chat / your IDE, you’ll get better results if you also tell it:[5]

- Which part to implement first (e.g., “Implement the React app structure and Tailwind theming according to sections 6, 9, and 10”).  
- That you want idiomatic, minimal dependencies (e.g., “Use React Router v6, no state libs beyond React state/hooks”).  
- That it should generate initial folder structure for `/apps`, `/suggestions`, `/logs`, plus one example app and `meta.json`.
